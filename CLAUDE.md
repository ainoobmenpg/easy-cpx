# CLAUDE.md

このファイルは、このリポジトリで Claude Code が常に参照する常設指示です。  
目的は、**MiniMax M2.5 を統裁AIとして利用する、ミリオタ向け作戦級CPX Webアプリ**を、破綻しにくい形で素早く実装することです。

---

## 0. このプロジェクトの要約

- プロダクト名: 仮称 `Operational CPX`
- 目的: プレイヤーが作戦級指揮官として命令を入力し、AI統裁とルールエンジンによって戦況が進行するCPXアプリを実装する
- 対象ユーザー: 軍事・兵器・冷戦/現代作戦に関心のある上級ホビー層
- 初版のコア体験:
  - 地図を見る
  - 命令を日本語で入力する
  - AIが命令を構造化する
  - ルールエンジンが裁定する
  - SITREPと地図更新を確認する

---

## 1. 最重要原則

このプロジェクトでは、**AIに全部を決めさせない**。

### 事実のソース・オブ・トゥルース
以下は必ず**アプリ側の状態管理 / DB / ルールエンジン**が保持すること。

- 地図
- ユニット位置
- 損耗状態
- 弾薬 / 燃料 / 整備余力
- 補給線
- 時刻
- 天候
- 観測済み情報
- 敵味方の既知情報差（Fog of War）
- 上官命令
- ターン履歴

### AIに任せること
- 日本語命令の解釈
- 曖昧命令の構造化
- SITREP文章化
- 上官命令文面の生成
- AAR要約
- UI内の説明文

### AIに任せないこと
- 真の敵配置の決定
- 位置更新の事実管理
- ダメージ計算本体
- 射程・視界・移動距離・補給線の本体判定
- プレイヤーが未観測の情報の漏洩

**文章は演出であり、事実は状態に従う。**

---

## 2. 開発スタンス

Claude Code は以下のスタンスで動くこと。

- できるだけ自走する
- 不要な確認質問を増やしすぎない
- ただし破壊的変更・仕様分岐・大きな設計変更の前は、短く意図を示す
- 作業前に関連ファイルを読み、局所最適ではなく全体整合を優先する
- 大きな変更は小さい差分に分割する
- 大きな変更は小さい差分に分割する
- **新規サービス作成時は必ず対応するテストファイルを一緒に作成する**
- 変更後は必ず関連テスト・lint・型チェックを回す
- 失敗したら、原因を特定してから最小修正で再試行する

### 判断方針
- 不明点があっても、既存仕様から妥当な仮定を置けるなら進める
- 仮定を置いた場合は、コメント・PR文・作業メモに明示する
- 「とりあえず動くが将来壊れる実装」より、少し遅くても拡張しやすい実装を選ぶ

## 3. プロダクト原則

### 3.1 CPXとしての原則
- 作戦級を主軸にする
- ナラティブより軍事的因果を優先する
- 情報は常に不完全である
- 成功は自動ではなく、部分成功・失敗・摩擦を普通に出す
- プレイヤーに敵の意図を露骨に教えない
- SITREPには、確認済み / 推定 / 不明 を分けて表現する

### 3.2 UX原則
- 画面を見れば「今なにが起きているか」が分かること
- 文章を読まなくても最低限プレイできること
- 命令テキストと、AIが解釈した構造化結果を両方見せること
- 地図表示とテキスト報告が矛盾しないこと

### 3.3 初版でやらないこと
- AI丸投げ裁定
- マルチプレイ
- 詳細な政治外交ゲーム
- 過度な3D演出
- 海空戦のフルシミュレーション

## 4. 技術方針

### 推奨スタック
- Frontend: Next.js / React / TypeScript
- Backend: Python / FastAPI
- DB: PostgreSQL
- State & fetching: Zustand / TanStack Query
- Map rendering: 初版はSVGまたはCanvas、必要なら将来 MapLibre GL JS

### コード品質
- TypeScript は `strict` 前提
- Python は型ヒントを基本とする
- 関数は短く、責務を明確にする
- ドメインモデルとUI都合の型を混同しない
- マジックナンバーを避け、定数化する

### ディレクトリ設計の原則
- `frontend/` と `backend/` を明確に分離
- `backend/domain/` にルールとモデルを集約
- `backend/services/ai/` にAI統合を集約
- `shared/` に JSON schema / 型定義 / 契約を置く
- `docs/` に仕様を集約

## 5. AI統合ルール

### 5.1 モデル前提
- 統裁AIは MiniMax M2.5 を想定する
- AI呼び出しは必ず**サーバー側**で行う
- APIキーはブラウザへ露出しない

### 5.2 AIロール分離
AIタスクは混ぜず、役割を分ける。

1. `order_parser`
   - 入力: プレイヤー自然文
   - 出力: 厳格な構造化JSON

2. `referee_writer`
   - 入力: 裁定結果JSON
   - 出力: SITREP本文

3. `excon_director`
   - 入力: 戦況サマリJSON
   - 出力: 上官命令更新 / 摩擦イベント候補 / 報告要求

### 5.3 AI出力のルール
- 自然文のままアプリ状態を更新しない
- AI出力は必ず schema で検証する
- 検証失敗時はリトライまたは安全なフォールバックを使う
- AI文面は、裁定結果JSONにない事実を付け足してはならない

### 5.4 プロンプト原則
- 長すぎる一般論を避ける
- 役割、入力、出力形式、禁止事項を明確化する
- 曖昧性は `assumptions` フィールドへ逃がす
- 日本語入力を前提にする

---

## 6. ルールエンジン原則

### 処理順
ターン処理は概ね以下の順で固定する。

1. 時間進行
2. 偵察 / 探知
3. 移動
4. 砲兵 / 間接火力
5. 直接戦闘
6. 防空 / 航空処理
7. 損耗更新
8. 弾薬 / 燃料 / 整備余力更新
9. 疲労 / 混乱更新
10. 摩擦イベント
11. 敵行動
12. 上官反応
13. SITREP生成

### 状態表現
初版では以下を優先する。

- 損耗: 無傷 / 軽損 / 中損 / 大損 / 戦闘不能
- 弾薬: 充足 / 低下 / 枯渇
- 燃料: 充足 / 低下 / 枯渇
- 整備余力: 充足 / 低下 / 枯渇
- 情報確度: 確認済み / 推定 / 未確認 / 不明

### 重要
- ルールは再現可能であること
- 同じ入力で概ね同じ結果になること
- ランダム要素がある場合はシード管理可能にすること

---

## 7. Fog of War ルール

- プレイヤー向けUIには、プレイヤーが知り得ない事実を出さない
- 敵の真位置とプレイヤー観測情報は別管理にする
- 地図描画は「表示用情報」から行い、「真実の情報」を直接使わない
- SITREPは観測済み情報のみを使う
- AARや管理者向けデバッグ表示では、閲覧権限を分離する

---

## 8. UI 実装ルール

### 必須画面領域
- 地図
- SITREP / 上官命令
- ユニット詳細
- 命令入力
- ターンログ

### UI原則
- 地図が主、文章は従
- 色やアイコンに意味を持たせる
- 推定情報と確認情報は見た目を分ける
- 命令送信前に、可能なら解釈プレビューを出す
- 1画面で現在ターンの重要情報にアクセスできるようにする

---

## 9. データ契約

Claude Code は、バックエンド / フロントエンド / AI の間で使う契約を必ず明示すること。

優先して作るもの:
- `shared/schemas/order-parser.schema.json`
- `shared/schemas/adjudication-result.schema.json`
- `shared/schemas/sitrep.schema.json`
- `shared/types/` の型定義

ルール:
- 先に schema を決めてから実装する
- 破壊的変更はバージョンを切る
- UI都合でレスポンスを曖昧にしない

## 10. テスト方針

### 必須テスト
- 命令解析テスト
- ルールエンジン単体テスト
- Fog of War 漏洩テスト
- API契約テスト
- 代表シナリオのゴールデンテスト

### 最低限回すコマンド
変更後は、関係するものを必ず回す。

- frontend lint
- frontend typecheck
- backend test
- backend lint / format
- schema validation tests

## 12. 実装優先順位

常に以下の優先順位を守ること。

1. 状態管理の正しさ
2. ルールエンジンの整合性
3. AI入出力の安定性
4. Fog of War の厳密性
5. UIの分かりやすさ
6. 演出の良さ

見た目の派手さより、まず**裁定が壊れないこと**を優先する。

---

## 13. 作業開始時の基本手順

新しい作業を始めるときは、原則この順で進めること。

1. 関連するコードと docs を読む
2. 既存の型 / schema / API 契約を確認する
3. 変更範囲を最小化する案を立てる
4. 実装する
5. テスト・lint・型チェックを回す
6. 必要なら docs を更新する
7. 変更内容と前提を簡潔にまとめる

---

## 14. 禁止事項

- AIの自然文をそのまま真実の戦況として保存しない
- プレイヤーに未観測情報を見せない
- スキーマなしの雑なAI連携を増やさない
- 巨大な単一ファイルに責務を詰め込まない
- 型エラー・lintエラーを放置しない
- 一時しのぎのハードコードを黙って残さない

---

## 15. 期待する出力スタイル

Claude Code の応答・作業は以下を意識すること。

- 簡潔
- 実装志向
- 仕様との関係が分かる
- 変更理由が明確
- 次に見るべきファイルや論点が追える

コード生成時は、**今このリポジトリに必要な最小十分の実装**を優先し、過剰設計を避ける。
